# Objetivo

Definir o perfil do operador, as tarefas principais, a jornada de uso e os princípios de interface que guiarão a implementação do frontend.

Este documento descreve o comportamento esperado do sistema sob a perspectiva do usuário, não a implementação técnica.

---

# Perfil do Operador

O operador do sistema possui o seguinte perfil:

- Único usuário do sistema
- Uso em campo, geralmente em pé
- Uso predominante em dispositivo móvel
- Operação rápida, com foco em execução (não exploração)
- Contexto: “resolver e sair rápido”

---

# Tarefas Principais

As tarefas do sistema estão organizadas por prioridade de acordo com a operação em campo:

| Prioridade | Tarefa |
|------------|--------|
| Crítica | Registrar pagamento de cliente |
| Crítica | Visualizar valor estimado do dia |
| Média | Cadastrar novo cliente |
| Média | Criar contrato |
| Média | Registrar gasto |
| Baixa | Consultar caixa semanal |
| Baixa | Visualizar clientes no mapa |

---

# Jornada Principal

Fluxo base de uso diário:

1. Abrir app → Dashboard (visão do dia)
2. Acessar cliente (lista ou busca rápida)
3. Ver pendências (parcelas / valores)
4. Registrar pagamento
5. Confirmar e retornar rapidamente ao contexto anterior  

---

# Princípios de Interface

A interface deverá respeitar os seguintes princípios:

- Uma ação principal por tela
- Evitar decisões múltiplas em campo
- Inputs grandes e fáceis de tocar
- Feedback imediato para toda ação
  - loading
  - sucesso
  - erro
- Informação crítica sempre visível:
  - valor pendente
  - valor recebido
  - saldo do cliente
- Navegação direta:
  - máximo de 2 toques para ação principal

---

# Core Task — Registro de Pagamento

Este é o fluxo mais importante do sistema e deve ser otimizado ao máximo.

Fluxo ideal:

Cliente → Parcelas → Toque em "Pagar" → Inserir valor → Confirmar

Regras:

- Deve levar menos de 10 segundos em condições normais
- Deve funcionar com mínima digitação possível
- Deve permitir confirmação rápida (sem telas intermediárias desnecessárias)

---

# Critério de Sucesso UX

O UX será considerado adequado quando:

- Registro de pagamento for feito em até 10s
- Nenhuma tarefa principal exigir mais de 3 telas
- Nenhuma ação principal exigir mais de 2 decisões
- Usuário não precisar “pensar onde clicar” na operação comum

# Contratos de Interface

Esta seção define o contrato de interface de cada módulo do sistema, especificando o conteúdo obrigatório, as ações permitidas e os estados de tela.

## Operações

### Tela: Central de Operações (Dashboard)

### Deve conter:
- Indicadores do dia (a receber, recebido, clientes para cobrar, resultado)
- Lista de cobranças do dia (por cliente/contrato)
- Seção de rota de cobrança com botão "Iniciar Rota"
- Indicador de GPS

### Deve permitir:
- Ver detalhes dos pagamentos do dia (modal)
- Iniciar rota de cobrança
- Navegar para contrato do cliente
- Contatar cliente (WhatsApp, Ligação)

### Não deve permitir:
- Registrar pagamento diretamente do dashboard

---

### Tela: Rota de Cobrança

### Deve conter:
- Card do cliente atual (nome, valor pendente, parcelas, distância)
- Badge de situação (Vence Hoje / Atrasado)
- Ações rápidas (Navegar, WhatsApp, Ligar, Abrir Contrato)
- Botão Pagar (ação principal)
- Ações de visita (Pular, Não Encontrado, Promessa)
- Navegação Anterior/Próximo

### Deve permitir:
- Registrar pagamento (modal com preview)
- Registrar visita (pular, não encontrado)
- Registrar promessa de pagamento (com data)
- Gerar e compartilhar comprovante (WhatsApp/compartilhar nativo)

### Não deve permitir:
- Avançar sem registrar ação no cliente atual

---

## Cliente

### Tela: Cliente (Detalhe)

### Deve conter:
- Nome do cliente
- Dados do cliente (comércio, CPF, telefone, endereço)
- Saldo total (situação financeira)
- Quantidade de contratos

### Deve permitir:
- Iniciar pagamento
- Acessar contratos
- Criar novo contrato
- Editar cliente

### Não deve permitir:
- Múltiplas ações simultâneas

---

### Tela: Lista de Clientes

### Deve conter:
- Busca por nome
- Lista de clientes (nome, comércio, telefone, cidade)

### Deve permitir:
- Buscar cliente
- Criar novo cliente
- Acessar detalhe do cliente

---

## Contrato

### Tela: Contrato (Detalhe)

### Deve conter:
- Nome do cliente (header)
- Resumo financeiro (saldo devedor, recebido, valor base, valor final, parcelas, datas, juros, status)
- Grade de parcelas com estado visual (Pendente, Parcial, Paga, Vencida)
- Histórico de pagamentos

### Deve permitir:
- Pagar parcela específica ou saldo total (modal com preview)
- Editar contrato (apenas se sem pagamentos)
- Excluir contrato (apenas se sem pagamentos, com confirmação)
- Navegar para o cliente

### Não deve permitir:
- Editar contrato com pagamentos registrados
- Excluir contrato com pagamentos registrados

---

### Tela: Lista de Contratos

### Deve conter:
- Filtro por cliente (dropdown com busca)
- Lista de contratos (cliente, capital, juros, saldo, total, parcelas, datas, status)
- Paginação

### Deve permitir:
- Filtrar por cliente
- Criar novo contrato
- Acessar detalhe do contrato

---

## Estados da Tela

### Loading
- Exibir skeleton que respeite a estrutura visual do conteúdo final

### Vazio
- Exibir mensagem objetiva + ação principal (ex: "Cadastrar cliente", "Novo contrato")

### Erro
- Exibir mensagem clara + botão de retry

### Sucesso
- Feedback global conforme definido em `plans/PLAN-004-feedback.md`

---

## Regras de Navegação

- Após pagamento → retornar para contexto anterior (cliente ou rota)
- Após criação → abrir entidade automaticamente
- Em erro → manter estado do formulário
- Máximo de 2 toques para ação principal
- Navegação deve preservar contexto do operador sempre que possível
